# 足場会社向け 信用プラットフォーム — 実装まとめ（v1〜v8 引き継ぎ資料）

> 別のAI（GPTなど）に共有して続きを相談するための資料です。プロトタイプの目的・思想・データモデル・業務ルール・実装済み機能・未実装事項・法務注意を、コードなしで把握できるようまとめています。実体は単一ファイルのReactプロトタイプ（`ashiba_platform_v8.jsx`）です。

---

## 1. プロダクトの目的と思想

- 足場会社グループ向けの**「信用（クレジット）プラットフォーム」**。求人マッチングサイトではなく、**「初めての相手とでも安全に取引できるよう、検証可能な"事実"を積み上げる場」**。
- 星評価やレビューではなく、**取引・支払い・入金・認証といった検証可能な事実**を蓄積して会社の信用にする。
- 役割分担：**募集の入口はLINE（コミュニティ）／正式な取引と記録はこのWebアプリ**。LINEを置き換えない。
- 「安全／危険」と断定せず、**確認できる事実（支払い実績、遅延件数、認証状況など）を並べ、取引判断は利用者自身が行う**。
- トラブル時、運営は**どちらか一方を悪者にせず、事実を記録して双方で解決を支援**する。

### デザイン方針（固定）
- 青×白、モバイルファースト、LINEライク、カードUI、大きなボタン、余白多め。
- 建設現場の職人が説明なしで使えること。
- 約390px幅のスマホフレーム（ヘッダー＋スクロール本文＋下部ナビ＋トースト）。

---

## 2. 技術構成

- **単一ファイルのReact（.jsx）**。外部CSSに依存せず、インラインstyle＋テーマオブジェクト `T`。`localStorage`不使用（すべて`useState`）。
- ルート`App`が全状態を保持：`companies / projects / txs / chats / nav / actingRole / lineConnected / lineCats / toast`。
- 画面遷移は`nav = {tab, screen, id, back}`で管理（ルーターライブラリ不使用）。
- **役割スイッチ（`actingRole` = 'prime'（元請）/ 'partner'（協力会社））**：プロトタイプで両者の操作を切り替えて体験できる。
- 検証方法（ネットワーク無効環境）：TypeScriptコンパイラでTSX構文チェック、正規表現でJSXコンポーネント参照チェック、状態機械のロジックをNodeで再実装してアサーション検証。

## 3. 画面構成

下部ナビ5タブ：**ホーム / 案件 / 取引 / パートナー / 自社**。

主な画面（`nav.screen`）：
- `home`：概要バナー、要対応の取引アラート（取引中／入金確認待ち／確認事項）、新着案件、登録会社一覧への導線。
- `projectList` / `projectDetail`：案件一覧・詳細（元請の信用、募集要項、応募者一覧）。
- `postJob`：案件投稿フォーム。
- `txList` / `txDetail`：取引管理（タブ：取引中／是正・手直し／請求・入金／確認事項／取引完了）と取引詳細。
- `chat`：案件専用チャット。
- `companyList` / `company`：登録会社一覧・会社プロフィール。
- `me`：自社プロフィール＋設定（登録会社一覧・LINE連携・認証管理・運営管理への導線）。
- `verify`：認証書類の提出・管理。
- `line`：LINE連携・通知カテゴリ設定・通知プレビュー。
- `partners`：資材/レンタル/保険/保証/金融/会計などのパートナー掲載（収益基盤）。
- `admin`：運営管理（デモ。統計、登録会社と信用レベル、確認事項のある取引、相談案件、信用レベル条件）。

## 4. データモデル

### 会社 company
`{ id, name, region, contact, areas, works, registered(登録日), verify(認証項目→状態), metrics }`
- `metrics = { completed(取引完了), paidCount(支払い完了), onTimeCount(期日内支払い), lateCount(遅延), unpaid, avgPayDays(平均支払日数), lastTrade, continuous(継続取引社数), _partners[] }`
- 認証`verify`の各項目状態：`none / reviewing / verified / expired / rejected`。
- 自社は `SELF_ID = "A"`（株式会社みらい足場）。他にB（東北ハウジング, Silver相当）、C（郡山スカイ, 未認証・登録7日以内）、D（仙台建装, Gold相当）。

### 案件 project
`{ id, stage('recruiting'|'matched'), name, jobType, region, address, start, end, assemblyStart, assemblyEnd, dismantleStart, dismantleEnd, need(nullable), price, payType, closing, payterm, work, belongings, deadline, posted, guaranteed, primeId, applicants[] }`

### 取引 transaction (tx)
`{ id, projectName, jobType, region, address, start, end, assemblyStart, assemblyEnd, dismantleStart, dismantleEnd, need(nullable), amount, payType, assemblyAmount, dismantleAmount, closing, payterm, primeId, partnerId, guaranteed, chatKey, order, orderAck, status(null|'completed'), ph, issues[], consultations[], ashibase, scheduleNotice, timeline[] }`
- `jobType`：`'support'（応援＝人工）` / `'contract'（請負＝一式）`。
- `payType`：`'progress'（出来高：組立分＋解体残金）` / `'lump'（一括：解体完了後にまとめて請求）`。
- `order / orderAck`：注文書 / 注文請書（`{at, date}`）。
- `scheduleNotice`：工期・予定変更の通知（`{at, changes[], ack}`）。
- `ashibase`：連携状態（`{linked, at}`）。
- `ph`：フェーズ状態（下記）。

### フェーズ ph.assembly / ph.dismantle（組立 / 解体）
`{ work, startDate, endDate, sessions[], report, rework, bill, inv, pay, dep }`
- `work`（作業トラック）：`waiting → working → reported →（rework）→ confirmed`
- `bill`（請求トラック）：`none → invoiced → checked → paid → deposited`
- `startDate / endDate`：実績の作業開始日 / 完了日。
- `sessions[]`：日次の作業記録 `{ date, kind('start'|'end'), people, content }`（複数日工期用。**ステータスを変えない**）。
- `report`：完了報告 `{ date, days, people, content, photos }`。
- `rework`：是正・手直し依頼 `{ text, date }`。
- `inv/pay/dep`：請求・支払い・入金確認の記録。

### チャット chats（案件専用・応募〜取引で継続）
`chats[key] = { primeId, partnerId, title, messages[] }`、`key = "案件ID:会社ID"`。応募者ごとに独立。依頼確定後に生成される取引が同じ`chatKey`を引き継ぎ、スレッドが継続する。

## 5. 取引エンジン（独立二相モデル）※最重要

組立と解体を**独立して進行**させる。従来の一本道（組立の入金完了まで解体に進めない）を廃止。

### 作業トラック（各フェーズ）
- `waiting`：作業開始待ち。**作業開始を報告**（開始日・応援なら人数を記録、`working`へ）。
- `working`：作業中。複数日工期なら**日次の作業開始／作業終了報告**が可能（`sessions`に記録、ステータスは不変）。**完了を報告**で`reported`へ。
- `reported`：完了報告済み・確認待ち。**元請**が「完了を確認」（`confirmed`）または「是正・手直しを依頼」（`rework`）。
- `rework`：是正・手直し中。**協力会社**が「是正・手直し完了」を押すまで先に進めない（押すと`reported`へ戻り、再度確認/是正が可能）。
- `confirmed`：作業完了確認済み。

**重要ルール**：解体作業の開始条件は「**組立作業がconfirmed**」であること**だけ**。組立分の請求・入金の完了は不要（＝組立分の入金前に解体日が到来しても解体を進められる）。

### 請求トラック（各フェーズ、作業confirmed後）
- `none → invoiced（協力が請求）→ checked（元請が請求確認）→ paid（元請が支払い済み登録）→ deposited（協力が入金確認）`
- **二者確認**：元請の「支払い済み」だけでは完了せず、協力の「入金確認」が必要。
- 入金額が請求額と異なると「**確認事項あり**」として登録され、先に進まない（当事者間または運営で解決）。
- 出来高：組立分＝`assemblyAmount`、解体分（残金）＝`dismantleAmount`。一括：組立分は請求せず、解体完了後に全額`amount`を請求。

### 完了条件と信用更新
- 出来高：組立・解体の両方が`deposited`で完了。一括：解体が`deposited`で完了。
- 完了時に`status='completed'`、期日内/遅延・平均支払日数を判定し、**双方の`metrics`を自動更新**（元請：completed/paidCount/onTime|late/avgPayDays/継続、協力：completed/継続）。

## 6. バージョン別の実装履歴（v1→v8）

- **v1**：MVP。ダッシュボード／案件／応募／チャット／マイページ、売掛保証バッジ、信頼スコア。
- **v2**：信用プラットフォーム化。二者による取引完了、信用タイムライン、支払い実績の自動集計、信用レベル（未認証/Bronze/Silver/Gold/Platinum 自動判定）、認証12項目、確認事項/安心材料、パートナー制度、LINE通知プレビュー、AshiBase連携、運営管理、役割スイッチ。
- **v3**：取引エンジンを組立→解体の二相（出来高/一括）に。元請の請求書確認ステップ、関係者限定の閲覧、登録会社一覧、運営への相談。
- **v4**：応援/請負の種別、募集人数を任意化、締め日＋支払日、**応募者ごとの案件専用チャット（成立後も継続）**、注文書/注文請書、**組立・解体の独立進行**、作業報告＋**是正・手直しループ**（元請の是正依頼／協力の是正完了）。
- **v5**：完了報告の人数は**応援のときだけ必須**、組立/解体に**開始・完了予定**を設定、作業の**開始報告と完了報告を分離**（実績の開始日/完了日）、**AshiBase連携ペイロード**（工程/勤怠/請求・原価/取引先/書類の正規化データ）。
- **v6**：複数日工期の場合に**作業終了報告（日次）**を追加（**ステータスを変えない**）、複数日案件では完了報告時に「**組立/解体が完了しましたか？**」の確認を表示。
- **v7**：取引詳細の各セクションを**折りたたみ（初期は非表示、タップで詳細）**に、完了報告の**のべ作業日数を自動計算**（日次報告＋完了日）、元請が**工期・組立/解体予定を変更**でき**協力へ通知**。
- **v8（最新）**：折りたたみで重要操作が隠れる問題を改善。**要対応の項目を自動展開＋ハイライト**、取引詳細上部に「**あなたの操作（要対応）**」を常時表示、**作業開始も日次で報告**可能（作業開始報告／作業終了報告、いずれもステータス不変・勤怠連携）。

## 7. 主要な業務ルール（現行=v8）

### 案件投稿（PostJob）
- 種別：応援／請負のトグル。**募集人数は任意**。
- 単価表記は種別で切替（応援＝日額/人工、請負＝請負金額 一式）。
- 支払条件：**締め日（末/25/20/15/10日締め）＋支払日（翌月末/翌月25/翌月15/翌々月末/当月末払い）**を選択。
- 支払方式：出来高／一括。
- **組立の開始・完了予定／解体の開始・完了予定**を個別に設定。
- 売掛保証トグル（表示のみ）。公開時にLINEへ新着通知（想定）。

### 案件詳細・応募者
- 元請の信用（レベル、期日内支払い、遅延、認証バッジ、確認できる事実）を提示。
- 応募者ごとに「**打合せチャット**」と「**この会社に依頼**」。依頼するとチャットを引き継いだ取引が生成される。

### 取引詳細（TxDetail）
- **関係者限定**（元請・協力・運営のみ閲覧）の注意表示。
- 役割スイッチ（元請/協力）で双方を体験。
- 現在のステータス表示（次にやること＝`nextHint`）。
- **「あなたの操作（要対応）」ハイライト（件数バッジ付き）を上部に常時表示**。取引開始/注文書・請書/作業開始・作業報告/完了確認・是正/請求・入金/確認事項/工期変更の確認 を集約。
- 各セクションは**折りたたみ**。ただし**自分が対応すべきもの（要対応）は自動で開き、アンバー色でハイライト＋「要対応」バッジ**。
- **契約書類**：元請が注文書を発行→協力が注文請書を発行（件名/種別/工期/組立・解体予定/金額/支払条件入りの書面プレビュー）。
- **組立 / 解体のフェーズパネル**：作業トラック（開始→日次報告→完了報告→確認/是正）＋請求トラック（請求→確認→支払→入金確認）。予定範囲・実績（開始/完了）・日次の開始終了ログを表示。
- **確認事項／運営への相談**：差額などを記録し双方で解決、運営へ事実確認を依頼。
- **案件情報**：元請は「編集」から**工期・組立/解体予定を変更**でき、変更すると協力側に通知バナー＋タイムライン記録。協力が「変更を確認しました」で消える。
- **AshiBase連携**：連携ON/再同期、連携できるデータ領域（工程/勤怠/請求・原価/取引先/書類）の一覧、**正規化ペイロード（JSON）の表示**。
- **信用タイムライン**：改ざん・削除できない記録として保存（想定）。

### 是正・手直しループ
元請が「是正・手直しを依頼」→ 協力が「是正・手直し完了」を押すまで、元請は完了確認に進めない。

### 複数日工期の作業報告
- 予定の開始≠完了（複数日）のフェーズでは、作業中に**作業開始を報告／作業終了を報告**（日次）が可能。**ステータスは変えない**。
- 完了報告時に「**◯◯は完了しましたか？**」の確認チェック（複数日のみ）。
- **のべ作業日数は自動計算**：日次報告＋完了日の実働日数（重複日は除外）。日次報告がない単日案件は手入力。

### AshiBase連携（設計）
- 取引データを施工管理側（AshiBase）へ渡せる**正規化ペイロード**として設計。領域は **工程（予定・実績日）／勤怠（作業開始・終了・人数）／請求・原価（契約金額・出来高内訳・締め/支払・請求/入金）／取引先（元請・協力）／書類（注文書・請書）**。
- ※ AshiBaseの実API仕様は未確認。実フィールド名・型・単位はAshiBase側に合わせてマッピングする前提（本接続は未実装）。

## 8. 信用レベル判定ロジック（自動）

会社の`metrics`と認証から自動判定（利用者は編集不可）。
- **未認証**：本人確認（電話/メール/法人番号/代表者/所在地）が未完了。
- **Bronze**：本人確認完了・取引1件以上。
- **Silver**：取引5件以上・期日内支払い90%以上・未解決の確認事項なし。
- **Gold**：取引20件以上・期日内95%以上・主要認証3項目以上（建設業許可/インボイス/労災/賠責）。
- **Platinum**：取引50件以上・期日内98%以上・遅延0・主要認証4項目。
- 条件は運営管理から変更でき、全社に再判定が反映される想定。

「確認できる事実」パネル：確認事項（本人確認未完了、支払い実績なし、登録7日以内、建設業許可未確認、保険未確認、遅延、未解決の確認事項）と、確認済み情報（本人確認済み、支払い実績、期日内支払い、遅延件数、継続取引、許可確認済み、保険確認済み）を並べる。

## 10. 法務上の注意（要・専門家確認）

- 制作者（AI）は弁護士ではない。以下は必ず弁護士・専門家に確認すること。
- **偽装請負／労働者派遣法**：個人単位のマッチングではなく、**会社対会社の請負構造**であることを担保する。
- **資金移動業／保険業／貸金業**：売掛保証・ファクタリング・決済・預り金は、本アプリでは**表示・導線のみ**で資金処理を実装していない。実装前に必ず確認。
- 「関係者限定の閲覧」「公開情報（信用レベル・支払い実績・認証）」の切り分けは、個人情報・営業秘密の観点でも要確認。

## 付録：GPTへ相談する際の観点例

- 独立二相エンジンの状態遷移に抜け漏れ・デッドロックがないか。
- 「要対応ハイライト／自動展開」のUXが現場職人に本当に分かりやすいか（情報過多になっていないか）。
- AshiBase連携ペイロードの項目設計（工程/勤怠/請求・原価/取引先/書類）が施工管理システム一般に接続しやすいか。
- 信用レベルの閾値・指標が、初期ユーザー数の少ない立ち上げ期でも機能するか。
- 法務リスク（偽装請負・資金移動業等）を回避する導線・契約フローになっているか。
